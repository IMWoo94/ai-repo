# [Feature] Operation Step Log for Logical Transaction Observability

## 작업 요약

충전/송금 처리 안의 주요 단계를 `operation_step_logs`로 기록한다. 현재는 모놀리식 애플리케이션이지만, 추후 MSA/Saga/Outbox 구조로 전환할 때 논리적 트랜잭션의 단계별 정합성과 관측 가능성을 설명할 수 있는 기반을 만든다.

## 배경과 문제

현재 저장소는 거래내역, 원장, 감사 이벤트를 기록한다. 이 기록들은 결과 중심 증거로는 충분하지만, 하나의 돈 이동 명령이 내부에서 어떤 단계를 거쳐 완료됐는지 추적하기에는 부족하다.

금융/핀테크 도메인에서는 하나의 프로세스가 여러 단계를 포함하더라도 다음 질문에 답할 수 있어야 한다.

- 어떤 단계까지 완료됐는가?
- 잔액 변경, 거래내역 기록, 원장 기록, 감사 기록, 멱등 기록의 순서는 무엇인가?
- 추후 서비스가 분리되면 어떤 단계가 Saga/Outbox/보상 처리 경계가 되는가?
- 장애 분석 시 결과 기록뿐 아니라 처리 과정 증거를 확인할 수 있는가?

## 작업 유형

관측 가능성 / 논리적 트랜잭션 기록

## 도메인 영역

충전, 송금, 원장, 감사, 정합성

## 범위

### 하는 것

- `OperationStepLog` 도메인 모델을 추가한다.
- 충전/송금 성공 처리 단계별 로그를 기록한다.
- PostgreSQL에는 `operation_step_logs` 테이블을 Flyway `V3` migration으로 추가한다.
- H2 빠른 저장소 테스트용 `schema.sql`도 같은 테이블을 반영한다.
- API에서 operation id 기준 step log를 조회한다.
- ADR, Wiki, issue draft에 MSA 전환 관점의 트레이드오프를 기록한다.

### 하지 않는 것

- Outbox publisher를 구현하지 않는다.
- Kafka 또는 메시지 브로커를 도입하지 않는다.
- 실패 단계 보상 처리나 재처리 워커를 구현하지 않는다.
- 서비스 분리를 실제로 수행하지 않는다.

## 도메인 규칙과 불변식

- step log는 원장 자체가 아니라 처리 과정 관측 기록이다.
- step log는 성공한 명령의 단계 완료 증거를 남긴다.
- 단계 로그 순서는 논리적 처리 순서와 일치해야 한다.
- 멱등 재시도는 기존 operation 결과를 반환하며 step log를 중복 생성하지 않는다.

## 대안과 트레이드오프

### 대안 A: 현재 감사 이벤트만 유지

장점:

- 구현이 단순하다.
- 기존 API와 저장소 변경이 없다.

단점:

- 내부 처리 단계의 관측 가능성이 부족하다.
- MSA 전환 시 Saga 단계나 Outbox 경계를 설명하기 어렵다.

### 대안 B: Operation Step Log 추가

장점:

- 모놀리스 안에서 논리적 트랜잭션의 단계별 증거를 남길 수 있다.
- 추후 Saga/Outbox 전환 시 단계 경계를 재사용할 수 있다.
- 금융 도메인에서 장애 분석과 정합성 설명력이 좋아진다.

단점:

- 기록 테이블과 테스트가 늘어난다.
- 아직 실패/보상 단계까지 표현하지 않으므로 후속 확장이 필요하다.

### 대안 C: Outbox/Saga를 즉시 도입

장점:

- MSA 전환에 가장 가까운 구조를 지금부터 검증할 수 있다.
- 메시지 발행, 재시도, 보상 처리 정책까지 함께 설계할 수 있다.

단점:

- 현재 모놀리스 학습 단계에 비해 범위가 크다.
- 브로커, outbox relay, idempotent consumer 등 부가 설계가 필요하다.

### 현재 선호안

Operation Step Log를 먼저 추가한다. 현재 단계의 목표는 서비스 분리가 아니라, 돈 이동 명령의 논리적 단계와 정합성 증거를 코드/테스트/문서로 남기는 것이다.
