# [Feature] 충전/송금 1차 흐름 구현

## 작업 요약

회원/지갑 계정 모델 위에 충전과 송금의 1차 API를 구현한다. 초기 범위는 인메모리 저장소, 동기 처리, 원화 단일 통화, 멱등키 기반 중복 요청 방지로 제한한다.

## 배경과 문제

잔액/거래내역 조회와 회원/지갑 계정 모델이 준비되었으므로 다음 단계는 실제 잔액을 변경하는 돈 이동 기능이다. 충전/송금은 핀테크 서비스의 핵심 위험인 잔액 정합성, 중복 요청, 계정 상태 검증을 드러내는 첫 기능이다.

현재 해결해야 할 문제는 다음과 같다.

- 잔액 변경을 수행하는 명령 서비스가 없다.
- 충전/송금 요청의 입력 검증 기준이 없다.
- 중복 요청이 들어왔을 때 잔액이 반복 반영될 위험이 있다.
- 거래내역이 조회 전용 샘플 데이터에 머물러 있다.

## 작업 유형

도메인 기능

## 도메인 영역

충전/송금

## 범위

### 하는 것

- 충전 API를 추가한다.
- 송금 API를 추가한다.
- 충전/송금 명령 서비스와 인메모리 저장소를 추가한다.
- 잔액 변경 시 거래내역을 함께 남긴다.
- 회원과 지갑이 모두 `ACTIVE`일 때만 돈 이동을 허용한다.
- 요청별 멱등키를 필수로 받아 중복 반영을 방지한다.
- 단위 테스트와 API 통합 테스트를 추가한다.

### 하지 않는 것

- 외부 계좌/PG 연동은 하지 않는다.
- 실제 인증/인가, KYC, AML은 구현하지 않는다.
- PostgreSQL 영속화와 트랜잭션 격리는 다음 단계로 미룬다.
- 분산락, 메시지 큐, 비동기 정산은 구현하지 않는다.
- 수수료, 한도, 환율은 구현하지 않는다.

## 수용 기준

- [ ] `POST /api/v1/wallets/{walletId}/charges`가 충전 요청을 처리한다.
- [ ] `POST /api/v1/wallets/{walletId}/transfers`가 송금 요청을 처리한다.
- [ ] 충전은 잔액을 증가시키고 거래내역에 `CHARGE/CREDIT/COMPLETED`를 남긴다.
- [ ] 송금은 출금 지갑 잔액을 감소시키고 입금 지갑 잔액을 증가시킨다.
- [ ] 송금은 출금 지갑에 `TRANSFER/DEBIT/COMPLETED`, 입금 지갑에 `TRANSFER/CREDIT/COMPLETED` 내역을 남긴다.
- [ ] 잔액 부족 송금은 실패한다.
- [ ] 비활성 회원 또는 비활성 지갑은 충전/송금할 수 없다.
- [ ] 같은 멱등키와 같은 요청은 잔액을 한 번만 변경한다.
- [ ] 같은 멱등키와 다른 요청은 실패한다.
- [ ] 문서와 ADR에 초기 트레이드오프가 기록되어 있다.

## 도메인 규칙과 불변식

- 충전 금액은 0보다 커야 한다.
- 송금 금액은 0보다 커야 한다.
- 송금 후 출금 지갑 잔액은 음수가 될 수 없다.
- 충전/송금은 원화 `KRW`만 지원한다.
- 충전/송금은 활성 회원의 활성 지갑에서만 가능하다.
- 모든 잔액 변경은 거래내역을 남긴다.
- 멱등키는 같은 요청의 재시도를 한 번의 결과로 수렴시킨다.

## 대안과 트레이드오프

### 대안 A: 충전만 먼저 구현

장점:

- 잔액 증가 흐름을 가장 작게 검증할 수 있다.
- 잔액 부족, 상대 지갑 검증, 양방향 거래내역을 미룰 수 있다.

단점:

- 핀테크 핵심인 지갑 간 돈 이동과 잔액 부족 검증이 늦어진다.
- 다음 단계에서 API와 서비스 경계를 다시 바꿀 가능성이 크다.

### 대안 B: 충전과 송금을 함께 구현

장점:

- 잔액 증가/감소와 거래내역 방향을 함께 검증할 수 있다.
- 멱등키, 잔액 부족, 계정 상태 검증의 기본 구조를 한 번에 세울 수 있다.
- 이후 원장/감사 로그 설계의 입력이 명확해진다.

단점:

- 첫 돈 이동 작업 범위가 커진다.
- 인메모리 저장소에서 동시성/트랜잭션 보장을 완전하게 표현하기 어렵다.

### 현재 선호안

충전과 송금을 함께 구현하되, 저장소는 인메모리 동기 처리로 제한한다. 멱등키는 1차부터 필수로 두어 중복 반영 위험을 조기에 테스트한다.
