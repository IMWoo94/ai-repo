# [Feature] 잔액 조회와 거래내역 조회 기준 정의

## 작업 요약

잔액 조회와 거래내역 조회를 첫 핀테크 기능 흐름으로 정의하고, 구현 전에 도메인 규칙, 수용 기준, 테스트 범위, 문서화 기준을 고정한다.

## 배경과 문제

이 레포의 첫 포트폴리오 증거물은 운영 하네스와 기능 흐름을 함께 보여줘야 한다. `잔액/거래내역`은 사용자가 금융 서비스처럼 즉시 이해할 수 있는 화면/기능이면서, 이후 `회원/지갑 계정`, `충전/송금`, `원장/감사 로그`로 확장하기 위한 기준점이다.

현재는 Spring Boot 애플리케이션 코드가 없으므로 바로 구현하기보다 다음을 먼저 고정해야 한다.

- 잔액과 거래내역의 최소 도메인 정의
- 조회 기능의 수용 기준
- 테스트 전략
- Wiki에 남길 사고 과정
- 이후 회원/지갑 계정, 충전/송금과의 연결 지점

## 작업 유형

도메인 기능

## 도메인 영역

잔액/거래내역

## 범위

### 하는 것

- 잔액 조회와 거래내역 조회의 도메인 용어를 정의한다.
- 잔액과 거래내역의 핵심 불변식을 정의한다.
- 조회 API가 제공해야 할 최소 응답 기준을 정의한다.
- 단위 테스트와 API 통합 테스트 범위를 정의한다.
- Wiki `Domain-Rules` 초안과 연결한다.
- 이후 구현 브랜치의 기준이 될 수용 기준을 작성한다.

### 하지 않는 것

- 실제 Spring Boot 애플리케이션 구현은 이번 Issue에서 하지 않는다.
- 인증/인가 구현은 이번 Issue에서 하지 않는다.
- 충전/송금으로 잔액을 변경하는 기능은 이번 Issue에서 하지 않는다.
- 원장 모델과 감사 로그의 최종 구조는 이번 Issue에서 확정하지 않는다.
- 외부 금융망, 실명 인증, 개인정보 처리는 포함하지 않는다.

## 도메인 규칙과 불변식

- 잔액은 특정 지갑의 현재 사용 가능 금액을 의미한다.
- 잔액은 음수가 될 수 없다.
- 거래내역은 잔액에 영향을 준 사건 또는 조회 가능한 금융 활동 기록이다.
- 거래내역은 감사 목적상 삭제하지 않는다.
- 거래내역 정렬 기본값은 최신순이다.
- 거래내역 조회는 빈 결과를 정상 결과로 취급한다.
- 잘못된 지갑 식별자 또는 존재하지 않는 지갑은 명확한 실패 응답을 반환해야 한다.
- 잔액 금액 단위와 통화는 명시되어야 한다.

## 대안과 트레이드오프

### 대안 A: 잔액/거래내역을 첫 기능으로 시작

장점:

- 사용자에게 금융 서비스처럼 보이는 결과를 빠르게 제공한다.
- 조회 기능이라 충전/송금보다 위험도가 낮다.
- 테스트와 문서화 흐름을 검증하기 좋다.

단점:

- 회원/지갑 계정 모델이 없으면 임시 식별자 또는 샘플 데이터가 필요하다.
- 실제 잔액 변경 원인이 아직 없으므로 거래내역 의미가 제한적이다.

### 대안 B: 회원/지갑 계정부터 시작

장점:

- 기반 모델을 먼저 안정화할 수 있다.
- 이후 잔액과 거래내역의 소유 관계가 명확해진다.

단점:

- 포트폴리오 관점에서 금융 서비스처럼 보이는 결과가 늦어진다.
- 사용자가 검증할 수 있는 기능 흐름이 덜 직관적이다.

### 현재 선호안

잔액/거래내역을 첫 기능 흐름으로 시작하되, 회원/지갑 계정과의 연결 지점을 문서에 명확히 남긴다.

## 릴리스 고려사항

- 실행 검증: 구현 단계에서는 최소 로컬 실행으로 잔액 조회와 거래내역 조회가 가능한지 확인한다.
- 마이그레이션: 초기 구현 전까지 없음.
- 알려진 리스크: 회원/지갑 계정 모델이 확정되기 전에는 식별자와 소유 관계가 임시일 수 있다.

## DECIDE_LATER

- 잔액을 `availableBalance`와 `ledgerBalance`로 분리할지 여부.
- 거래내역과 원장 항목을 같은 모델로 볼지 분리할지 여부.
- 통화 단위를 원화로 고정할지 다중 통화를 고려할지 여부.
- 인증/인가가 도입되기 전 조회 권한을 어떻게 표현할지 여부.
- 첫 구현에서 DB를 PostgreSQL로 바로 사용할지, 테스트 편의를 위해 H2를 병행할지 여부.
