# [Feature] 회원/지갑 계정 모델 정의와 조회 API 연결

## 작업 요약

잔액/거래내역 조회 API의 임시 `wallet-001` 기준을 회원/지갑 계정 모델로 확장하고, 지갑 소유 관계와 계정 생명주기 기준을 정의한다.

## 배경과 문제

Issue #1에서는 첫 기능 흐름을 빠르게 검증하기 위해 인메모리 지갑과 거래내역을 사용했다. 다음 단계에서는 조회 기능이 실제 도메인 기반 위에 올라가도록 회원과 지갑 계정의 관계를 정의해야 한다.

현재 남아 있는 문제는 다음과 같다.

- 지갑이 누구에게 속하는지 명확하지 않다.
- 회원과 지갑의 생성/활성/비활성 상태가 없다.
- 존재하지 않는 지갑과 접근 권한 없는 지갑을 구분할 기준이 없다.
- 이후 충전/송금으로 확장할 때 돈 이동 주체가 불명확하다.

## 작업 유형

도메인 기능

## 도메인 영역

회원/지갑 계정

## 범위

### 하는 것

- 회원과 지갑 계정의 최소 도메인 모델을 정의한다.
- 회원과 지갑의 1차 관계를 정의한다.
- 지갑 상태 후보를 정의한다.
- 기존 잔액/거래내역 조회 API가 지갑 계정 모델과 연결되도록 개선한다.
- 인메모리 저장소를 Repository 역할로 분리한다.
- 단위 테스트와 API 통합 테스트를 보강한다.

### 하지 않는 것

- 실제 로그인/인증/인가 구현은 하지 않는다.
- 개인정보, 실명 인증, KYC는 다루지 않는다.
- 충전/송금 기능은 구현하지 않는다.
- PostgreSQL 영속화는 다음 단계로 미룬다.
- 원장 모델은 확정하지 않는다.

## 수용 기준

- [ ] `Member`, `WalletAccount` 또는 동등한 도메인 모델이 정의되어 있다.
- [ ] 지갑은 소유 회원 식별자를 가진다.
- [ ] 지갑 상태가 문서와 코드에 표현되어 있다.
- [ ] 비활성 지갑 조회 정책이 명시되어 있다.
- [ ] 기존 잔액/거래내역 조회 테스트가 유지된다.
- [ ] 회원/지갑 계정 도메인 테스트가 추가된다.
- [ ] 인메모리 데이터 접근이 서비스에서 분리되어 있다.
- [ ] API 오류 응답이 기존 형식과 일관된다.

## 도메인 규칙과 불변식

- 회원은 0개 이상의 지갑을 가질 수 있다.
- 초기 학습 범위에서는 회원당 기본 지갑 1개를 우선 가정한다.
- 지갑은 반드시 소유 회원 식별자를 가진다.
- 지갑 상태는 최소 `ACTIVE`, `SUSPENDED`, `CLOSED` 후보를 가진다.
- 닫힌 지갑은 잔액/거래내역 조회 정책을 별도 결정해야 한다.
- 지갑 식별자는 전역에서 유일해야 한다.

## 대안과 트레이드오프

### 대안 A: 회원당 지갑 1개로 시작

장점:

- 모델이 단순하다.
- 잔액/거래내역 조회와 연결하기 쉽다.
- 다음 충전/송금 단계로 빠르게 이동할 수 있다.

단점:

- 실제 핀테크 서비스의 복수 계좌/복수 지갑 구조를 바로 표현하지 못한다.

### 대안 B: 회원당 지갑 N개를 처음부터 지원

장점:

- 확장성이 좋다.
- 다중 지갑, 목적별 지갑, 통화별 지갑을 표현하기 쉽다.

단점:

- 첫 학습 단계에서 범위가 커진다.
- API와 테스트 케이스가 늘어난다.

### 현재 선호안

회원은 0개 이상의 지갑을 가질 수 있는 모델로 설계하되, 초기 샘플 데이터와 API는 기본 지갑 1개 흐름으로 제한한다.
