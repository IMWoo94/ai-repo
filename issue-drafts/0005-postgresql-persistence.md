# [Feature] PostgreSQL 저장소와 멱등키/원장 영속화 1차 도입

## 작업 요약

인메모리 저장소에 있는 회원, 지갑, 잔액, 거래내역, 멱등키, 원장, 감사 로그를 PostgreSQL 스키마로 옮길 수 있는 JDBC 저장소를 추가한다. 기본 실행은 인메모리로 유지하고, `postgres` 프로필에서 JDBC 저장소를 사용한다.

## 배경과 문제

현재 기능은 충전/송금, 원장, 감사 로그까지 인메모리로 동작한다. 하지만 프로세스 재시작 시 잔액, 멱등키, 원장, 감사 로그가 모두 초기화된다. 핀테크 서비스에서 멱등성, 원장, 감사 기록은 프로세스 수명과 분리되어 영속화되어야 한다.

현재 남아 있는 문제는 다음과 같다.

- 멱등키 기록이 재시작 후 사라진다.
- 원장과 감사 로그가 재시작 후 사라진다.
- 잔액 변경을 DB 트랜잭션으로 묶을 기준 구현이 없다.
- PostgreSQL 스키마와 애플리케이션 도메인 모델의 매핑 기준이 없다.

인프라 기능

## 도메인 영역

저장소/원장/멱등성

## 범위

### 하는 것

- Spring JDBC 의존성을 추가한다.
- PostgreSQL 프로필 설정을 추가한다.
- PostgreSQL용 스키마 SQL을 추가한다.
- JDBC 기반 지갑 저장소를 추가한다.
- 멱등키, 거래내역, 원장, 감사 로그를 DB에 저장하도록 구현한다.
- H2 기반 JDBC 저장소 테스트를 추가해 SQL 매핑을 검증한다.
- ADR과 Wiki에 인메모리 저장소와 PostgreSQL 저장소의 경계를 기록한다.

### 하지 않는 것

- 운영용 Docker Compose는 이번 단계에서 필수로 두지 않는다.
- Fly.io 배포는 이번 단계에서 구현하지 않는다.
- 고급 동시성 제어, 분산락, 격리 수준 튜닝은 후속 단계로 미룬다.
- DB 마이그레이션 도구 Flyway/Liquibase 도입은 별도 결정으로 남긴다.
- 개인정보, 인증/인가 저장소는 구현하지 않는다.

- 멱등키는 영속 저장되어야 한다.

- 원장 엔트리는 append-only를 목표로 한다.

## 대안과 트레이드오프

### 대안 A: PostgreSQL을 바로 기본 저장소로 전환

장점:

- 실제 운영 환경에 가까워진다.
- 인메모리와 DB 저장소의 이중 경로가 줄어든다.

단점:

- 로컬 개발과 CI가 DB 의존성을 즉시 갖게 된다.
- 아직 Docker/Testcontainers/마이그레이션 전략이 고정되지 않았다.
- 학습 진행 속도가 느려질 수 있다.

### 대안 B: 기본은 인메모리, `postgres` 프로필에서 JDBC 저장소 사용

장점:

- 기존 로컬 실행과 CI 안정성을 유지한다.
- DB 저장소를 점진적으로 검증할 수 있다.
- PostgreSQL 전환 리스크를 ADR과 테스트로 분리할 수 있다.

단점:

- 저장소 구현이 두 개 유지된다.
- H2 테스트와 PostgreSQL 실제 동작 차이가 있을 수 있다.
- 운영 전환 시 프로필 설정 검증이 필요하다.

### 현재 선호안

기본은 인메모리로 유지하고, `postgres` 프로필에서 JDBC 저장소를 사용한다. 저장소 테스트는 H2로 빠르게 검증하고, 실제 PostgreSQL 실행 검증은 다음 단계에서 Docker Compose 또는 Testcontainers 도입과 함께 강화한다.

## 릴리스 고려사항

- 실행 검증: 기본 프로필 `./gradlew check`, `./gradlew bootRun`.
- DB 실행 검증: 이번 단계에서는 H2 저장소 테스트로 대체한다.
- 마이그레이션: SQL 파일은 제공하지만 자동 마이그레이션 도구는 아직 없다.
- 알려진 리스크: H2와 PostgreSQL SQL 방언 차이가 남아 있다.
