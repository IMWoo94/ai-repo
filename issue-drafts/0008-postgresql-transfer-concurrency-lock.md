# [Feature] PostgreSQL 송금 동시성 잠금 정책 검증

## 작업 요약

PostgreSQL 저장소에서 충전/송금 잔액 갱신 시 row-level lock을 사용하고, 송금 잔액 부족 검증을 트랜잭션 내부에서 다시 수행한다. Testcontainers PostgreSQL 테스트로 동시 송금 상황에서 잔액이 음수가 되지 않고 원장/감사 로그가 성공 거래만 남는지 검증한다.

## 배경과 문제

Issue #9, #11, #13을 통해 PostgreSQL 저장소, 런타임 검증, Flyway migration을 도입했다. 하지만 현재 송금 흐름은 서비스 계층에서 잔액 부족을 먼저 확인한 뒤 저장소 트랜잭션에서 잔액을 갱신한다.

이 구조는 단일 JVM의 `synchronized` 서비스 인스턴스에서는 단순하게 동작하지만, 실제 운영 환경처럼 여러 애플리케이션 인스턴스나 여러 서비스 객체가 같은 DB를 갱신하면 다음 문제가 생길 수 있다.

- 두 송금 요청이 같은 출금 지갑 잔액을 동시에 읽는다.
- 두 요청 모두 잔액 충분으로 판단한다.
- 저장소 트랜잭션에서 마지막 update가 앞선 update를 덮어쓰거나 음수 방지 불변식을 우회한다.
- 원장/감사 로그가 실제 잔액과 불일치할 수 있다.

## 작업 유형

저장소 안정성 / 동시성 검증

## 도메인 영역

송금, 잔액, 원장, 감사 로그

## 범위

### 하는 것

- PostgreSQL JDBC 저장소의 잔액 조회/갱신 경로에 `SELECT ... FOR UPDATE` 기반 row-level lock을 적용한다.
- 송금 잔액 부족 검증을 저장소 트랜잭션 내부에서 다시 수행한다.
- 송금 시 두 지갑을 결정적 순서로 잠가 deadlock 가능성을 줄인다.
- Testcontainers PostgreSQL 테스트에 동시 송금 검증을 추가한다.
- ADR, Wiki, issue draft에 동시성 정책과 트레이드오프를 기록한다.

### 하지 않는 것

- 분산락을 도입하지 않는다.
- Kafka/event sourcing을 도입하지 않는다.
- 낙관적 락 version column을 추가하지 않는다.
- 격리 수준을 전역 `SERIALIZABLE`로 올리지 않는다.

## 수용 기준

- [ ] 충전은 잔액 row를 lock한 뒤 갱신한다.
- [ ] 송금은 출금/입금 지갑 잔액 row를 결정적 순서로 lock한 뒤 갱신한다.
- [ ] 송금 잔액 부족은 트랜잭션 내부 locked balance 기준으로 재검증한다.
- [ ] 동시 송금 2건 중 잔액을 초과하는 요청은 하나만 성공하고 나머지는 실패한다.
- [ ] 실패한 송금은 거래내역, 원장, 감사 로그, 멱등 기록을 남기지 않는다.
- [ ] `./gradlew check`가 통과한다.
- [ ] 문서에 DB row lock 선택 근거와 대안이 기록되어 있다.

## 도메인 규칙과 불변식

- 송금 후 출금 지갑 잔액은 음수가 될 수 없다.
- 성공한 송금만 거래내역, 원장, 감사 로그, 멱등 기록을 남긴다.
- 같은 지갑 쌍을 동시에 송금해도 lock 획득 순서는 결정적이어야 한다.

## 대안과 트레이드오프

### 대안 A: 현재 서비스 `synchronized` 유지

장점:

- 코드 변경이 적다.
- 단일 JVM 데모에서는 단순하다.

단점:

- 여러 인스턴스나 여러 서비스 객체가 같은 DB를 쓰는 운영 유사 환경을 보호하지 못한다.
- DB 정합성을 애플리케이션 객체 락에 의존한다.

### 대안 B: PostgreSQL row-level lock

장점:

- 정합성 기준을 DB 트랜잭션 경계에 둔다.
- 특정 지갑 잔액 row만 잠가 범위를 작게 유지한다.
- 현재 JDBC 저장소 구조에 가장 적게 침투한다.

단점:

- 경합 지갑에서는 대기 시간이 늘 수 있다.
- lock 순서 정책을 지키지 않으면 deadlock 위험이 있다.

### 대안 C: 낙관적 락 version column

장점:

- 충돌이 낮은 경우 lock 대기 비용이 작다.
- 충돌 감지와 재시도 정책을 명시적으로 설계할 수 있다.

단점:

- 스키마 변경이 필요하다.
- 재시도 정책, 오류 응답, 사용자 경험 설계가 함께 필요하다.

### 현재 선호안

PostgreSQL row-level lock을 선택한다. 현재 단계의 핵심은 금융 잔액 불변식을 DB 트랜잭션 안에서 지키는 것이며, 스키마 변경 없이 Testcontainers로 검증할 수 있다.
