# Harness Roles

이 문서는 `ai-repo`에서 사용하는 하네스 역할의 책임, 산출물, 체크 기준을 설명한다.

역할은 실제 사람을 대체하는 것이 아니라, 작업을 여러 관점으로 검증하기 위한 구조다. 사용자는 코드 개발자로 직접 참여하며 최종 판단권을 가진다.

## 운영 원칙

- 초기에는 모든 역할을 촘촘히 적용한다.
- 반복 패턴이 안정되면 체크리스트 중심으로 경량화한다.
- 돈 이동, 원장, 동시성, 멱등성, 정산 같은 고위험 기능은 경량화하지 않는다.
- 중요한 결정은 ADR에 남긴다.
- 긴 사고 과정과 역할별 논의는 Wiki에 남긴다.
- PR에는 요약과 링크를 남긴다.

## 역할 요약

| 역할 | 핵심 책임 | 주요 산출물 |
| --- | --- | --- |
| 사용자 개발자 | 최종 판단과 구현 참여 | 코드, 리뷰 의견, 승인/보류 결정 |
| 기획자 | 문제와 완료 기준 정의 | Issue, 사용자 시나리오, 수용 기준 |
| 도메인 전문가 | 금융 규칙과 위험 검토 | 도메인 규칙, 불변식, 예외 정책 |
| 코드 개발자 A | 구현과 테스트 작성 | 코드, 단위 테스트, 통합 테스트 |
| 코드 개발자 B | 대안과 결함 검토 | 리뷰 코멘트, 구조 개선안 |
| QA | 실패 조건과 회귀 검증 | QA 체크리스트, 경계값, 테스트 시나리오 |
| 릴리스 관리자 | 실행 가능한 결과물 관리 | 릴리스 노트, 실행 방법, 알려진 리스크 |

## 사용자 개발자

### 책임

- 최종 목표와 범위를 승인한다.
- 구현에 직접 참여한다.
- AI 역할들의 제안을 검증하고 채택 여부를 결정한다.
- 문서와 코드가 실제 학습 목표에 맞는지 판단한다.

### 체크 기준

- 문서를 읽고 작업 목적을 설명할 수 있는가?
- 구현이 도메인 규칙을 위반하지 않는가?
- 테스트가 충분한가?
- 포트폴리오 증거물로 설명 가능한가?

### 할 수 없는 것

- 테스트 없는 코드 변경을 완료로 승인할 수 없다.
- ADR이 필요한 결정을 구두 결정으로만 넘길 수 없다.
- 문서와 코드의 모순을 방치할 수 없다.

## 기획자

### 책임

- 문제를 사용자 관점에서 정의한다.
- 이번 작업에서 하는 것과 하지 않는 것을 분리한다.
- 수용 기준을 체크 가능한 문장으로 작성한다.

### 산출물

- Issue의 배경과 문제
- 사용자 시나리오
- 수용 기준
- 범위와 제외 범위

### 체크 질문

- 이 작업은 왜 지금 필요한가?
- 사용자는 무엇을 보고 성공을 판단하는가?
- 이번 작업에서 의도적으로 하지 않는 것은 무엇인가?
- 수용 기준은 테스트 또는 리뷰로 확인 가능한가?

## 도메인 전문가

### 책임

- 금융 도메인 규칙과 불변식을 정의한다.
- 예외 상황과 실패 정책을 검토한다.
- 구현 편의 때문에 도메인 규칙이 약해지지 않도록 한다.

### 산출물

- 도메인 규칙 목록
- 상태 전이와 예외 정책
- 금액, 잔액, 거래내역, 원장 관련 위험 지적
- ADR 또는 Wiki에 남길 도메인 결정 후보

### 체크 질문

- 잔액이 음수가 될 수 있는가?
- 거래내역은 수정 또는 삭제 가능한가?
- 중복 요청은 어떻게 처리되는가?
- 실패한 거래는 어떤 상태로 남는가?
- 감사 목적으로 추적해야 하는 정보는 무엇인가?

## 코드 개발자 A

### 책임

- 합의된 범위에 맞춰 구현한다.
- 테스트 코드를 함께 작성한다.
- PR에 구현 의도와 테스트 결과를 기록한다.

### 산출물

- 구현 코드
- 단위 테스트
- 통합 테스트
- PR 설명

### 체크 질문

- 코드가 Issue 수용 기준을 직접 만족하는가?
- 도메인 규칙이 코드에 표현되어 있는가?
- 실패 케이스 테스트가 있는가?
- 구현이 과도하게 복잡하지 않은가?

## 코드 개발자 B

### 책임

- 코드 개발자 A의 구현을 독립적으로 리뷰한다.
- 대안과 트레이드오프를 제시한다.
- 장기 유지보수성과 구조적 리스크를 검토한다.

### 산출물

- 리뷰 코멘트
- 구조 개선안
- 테스트 누락 지적
- ADR 필요 여부 의견

### 체크 질문

- 더 단순한 구현이 가능한가?
- 테스트가 구현 세부사항에 과하게 결합되어 있지 않은가?
- 도메인 모델과 API 경계가 명확한가?
- 이 구조가 다음 기능으로 확장 가능한가?

## QA

### 책임

- 수용 기준을 검증 가능한 시나리오로 바꾼다.
- 경계값, 실패 조건, 회귀 위험을 확인한다.
- 릴리스 후보의 실행 가능성을 점검한다.

### 산출물

- QA 체크리스트
- 정상/예외 시나리오
- 경계값 목록
- 회귀 테스트 필요 여부

### 체크 질문

- 빈 결과, 잘못된 입력, 권한 없음, 중복 요청은 검증했는가?
- 잔액, 거래내역, 상태 전이가 기대대로 남는가?
- 실패 시 사용자와 시스템은 어떤 상태를 보는가?
- 버그 수정이면 재현 테스트가 먼저 있는가?

## 릴리스 관리자

### 책임

- 변경을 버전 단위로 묶을 수 있는지 판단한다.
- 실행 방법과 테스트 결과를 릴리스 노트에 남긴다.
- 알려진 리스크와 다음 작업 후보를 정리한다.

### 산출물

- 릴리스 노트
- 실행 검증 결과
- 알려진 리스크
- 다음 릴리스 후보

### 체크 질문

- 로컬에서 실행 가능한가?
- 어떤 테스트를 실행했는가?
- 사용자가 릴리스 결과를 어떻게 확인하는가?
- 알려진 리스크가 문서화되었는가?

## 역할 적용 강도

| 단계 | 적용 방식 |
| --- | --- |
| 초기 | 모든 역할의 사고 과정을 상세히 기록 |
| 반복 안정 후 | 반복 항목은 체크리스트로 경량화 |
| 고위험 기능 | 상세 기록 유지 |
| 문서 전용 작업 | 코드 개발자/QA 항목은 문서 검증 중심으로 축소 |

## PR 기록 방식

PR에는 역할별 전체 사고 과정을 모두 쓰지 않는다.

PR에는 다음만 남긴다.

- 역할별 최종 판단 요약
- 차단 이슈
- 테스트 결과
- 관련 ADR 링크
- 관련 Wiki 링크

상세 사고 과정은 Wiki에 남긴다.

## 역할별 완료 기준

| 역할 | 완료 기준 |
| --- | --- |
| 기획자 | Issue의 배경, 범위, 수용 기준이 검증 가능함 |
| 도메인 전문가 | 도메인 규칙과 예외가 누락 없이 검토됨 |
| 코드 개발자 A | 코드와 테스트가 수용 기준을 만족함 |
| 코드 개발자 B | 구조, 대안, 테스트 누락을 검토함 |
| QA | 정상/예외/경계값/회귀 위험을 검토함 |
| 릴리스 관리자 | 실행 방법, 테스트 결과, 리스크가 기록됨 |

## 금지 사항

- 테스트 없는 기능 완료 처리
- ADR 없이 중요한 결정 확정
- Wiki에만 기준 결정 기록
- PR에 테스트 결과 미기재
- 도메인 규칙을 구현 편의로 약화
- 릴리스 노트 없는 릴리스 완료
