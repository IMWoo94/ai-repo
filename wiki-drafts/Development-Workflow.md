# Development Workflow

이 문서는 `ai-repo`에서 기능 작업을 시작하고 검증하고 릴리스하는 흐름을 설명한다.

중요한 결정의 기준은 ADR이다.

- ADR-0001: Documentation Source of Truth
- ADR-0002: Test Strategy

## 목표

작업 흐름의 목표는 다음 세 가지다.

1. 기능 개발 전에 문제와 완료 기준을 명확히 한다.
2. 코드, 테스트, 문서, 리뷰, QA, 릴리스를 하나의 흐름으로 연결한다.
3. 역할별 사고 과정과 검증 결과를 포트폴리오 증거물로 남긴다.

## 전체 흐름

```text
아이디어
  → 인터뷰/스펙
  → Issue
  → ADR 필요 여부 판단
  → 기능 브랜치
  → 구현 + 테스트
  → PR
  → 역할별 리뷰
  → QA 검증
  → 릴리스 후보
  → 실행 검증
  → Release
  → 회고/Wiki 갱신
```

## 1. 아이디어와 인터뷰

작업 주제가 모호하면 바로 구현하지 않는다.

먼저 다음을 확인한다.

- 무엇을 만들 것인가?
- 왜 지금 필요한가?
- 사용자는 무엇을 검증할 수 있어야 하는가?
- 이번 작업에서 하지 않을 것은 무엇인가?
- 코드와 테스트 외에 어떤 문서가 필요한가?

모호성이 크면 `deep-interview` 또는 `interview` 스킬을 사용한다.

## 2. 스펙 작성

중요한 기능 또는 운영 기준은 스펙 문서로 먼저 고정한다.

스펙에는 최소한 다음을 포함한다.

- 배경과 문제
- 사용자와 목표
- 정상 흐름과 예외 상황
- 주요 데이터와 상태
- 연결되는 시스템
- 권한
- 범위와 성공 기준
- 개발 정책

스펙은 검증 가능한 언어로 작성한다. “잘 동작한다” 같은 표현은 완료 기준으로 충분하지 않다.

## 3. Issue 생성

작업은 GitHub Issue로 시작한다.

Issue 템플릿은 `.github/ISSUE_TEMPLATE/feature.yml`을 사용한다.

Issue에는 다음이 있어야 한다.

- 작업 요약
- 배경과 문제
- 작업 유형
- 도메인 영역
- 하는 것과 하지 않는 것
- 수용 기준
- 도메인 규칙과 불변식
- 하네스 역할 체크
- 예상 테스트 범위
- 문서화 필요 여부
- 대안과 트레이드오프
- 릴리스 고려사항

## 4. ADR 필요 여부 판단

다음에 해당하면 ADR을 작성한다.

- 기술 선택이 바뀐다.
- 테스트 전략이 바뀐다.
- 문서 기준이나 운영 흐름이 바뀐다.
- 도메인 정책이나 불변식이 바뀐다.
- 릴리스 완료 기준이 바뀐다.
- 장기 유지보수에 영향을 주는 구조 결정이 있다.

ADR이 필요하지 않다면 PR에 “ADR 불필요 사유”를 남긴다.

## 5. 브랜치 생성

브랜치 규칙은 다음을 따른다.

| 유형 | 규칙 |
| --- | --- |
| 기능 | `feature/<issue-number>-<short-name>` |
| 버그 수정 | `fix/<issue-number>-<short-name>` |
| 문서 | `docs/<issue-number>-<short-name>` |
| 릴리스 | `release/v<major>.<minor>` |
| 긴급 수정 | `hotfix/v<major>.<minor>.<patch>` |

브랜치는 Issue와 연결되어야 한다.

## 6. 구현과 테스트

코드 변경이 있으면 테스트 코드가 필요하다.

테스트 기준은 ADR-0002를 따른다.

| 변경 유형 | 최소 테스트 |
| --- | --- |
| 도메인 규칙 | 단위 테스트 |
| API 추가/변경 | API 통합 테스트 |
| Repository/DB 변경 | Repository 통합 테스트 |
| 잔액/돈 이동 | 동시성 또는 멱등성 테스트 계획 |
| 버그 수정 | 실패 재현 회귀 테스트 |
| 릴리스 대상 | 로컬 실행 검증 |

프로젝트 스캐폴딩 이후 기본 명령은 다음으로 둔다.

```bash
./gradlew test
./gradlew check
```

## 7. PR 생성

PR 템플릿은 `.github/pull_request_template.md`를 사용한다.

PR에는 다음을 기록한다.

- 변경 내용
- 관련 Issue
- 관련 ADR
- 관련 Wiki
- 역할별 검토 결과
- 테스트 실행 결과
- 도메인 규칙 영향
- 대안과 트레이드오프
- 릴리스 고려사항

PR은 단순 코드 제출이 아니라 변경 검증 문서다.

## 8. 역할별 리뷰

초기 단계에서는 모든 역할을 촘촘하게 적용한다.

필수 관점:

- 기획자
- 도메인 전문가
- 코드 개발자 A
- 코드 개발자 B
- QA
- 릴리스 관리자

패턴이 안정되면 반복적인 항목은 체크리스트 중심으로 줄인다. 단, 돈 이동, 원장, 동시성, 멱등성, 정산 같은 고위험 기능은 계속 상세 리뷰를 유지한다.

## 9. QA 검증

QA는 다음을 확인한다.

- 수용 기준이 모두 충족되었는가?
- 경계값이 검증되었는가?
- 실패 시나리오가 검증되었는가?
- 회귀 테스트가 필요한가?
- 문서와 코드가 모순되지 않는가?
- 릴리스 후보라면 실행 검증이 되었는가?

## 10. 릴리스

릴리스는 단순 태그가 아니다.

릴리스 노트에는 다음을 포함한다.

- 포함된 기능
- 변경된 도메인 규칙
- 테스트 결과
- 실행 방법
- 마이그레이션 필요 여부
- 알려진 리스크
- 다음 릴리스 후보

초기에는 로컬 실행 검증을 최소 기준으로 둔다. Fly.io 등 외부 배포는 별도 ADR 또는 릴리스 기준 변경 후 도입한다.

## 11. 회고와 Wiki 갱신

작업이 끝나면 Wiki에 사고 과정과 검증 결과를 남긴다.

남길 내용:

- 어떤 대안이 있었는가?
- 왜 현재 결정을 선택했는가?
- 어떤 역할이 어떤 리스크를 제기했는가?
- 테스트에서 무엇을 확인했는가?
- 다음 작업에서 개선할 점은 무엇인가?

## 완료 체크리스트

- [ ] Issue에 수용 기준이 있다.
- [ ] ADR 필요 여부를 판단했다.
- [ ] 기능 브랜치가 Issue와 연결된다.
- [ ] 코드 변경이 있으면 테스트가 있다.
- [ ] 역할별 리뷰 결과가 PR에 있다.
- [ ] QA 검증 결과가 있다.
- [ ] 릴리스 대상이면 실행 검증과 릴리스 노트가 있다.
- [ ] 긴 사고 과정은 Wiki에 기록된다.
