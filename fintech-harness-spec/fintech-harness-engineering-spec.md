# 핀테크 하네스 엔지니어링 — 기획서

## 1. 배경

이 저장소는 Java 25와 Spring Boot 기반으로 핀테크 서비스를 학습하기 위한 공간이다. 단순히 API를 구현하는 것이 아니라, 금융권 서비스 개발에서 중요한 정합성, 감사 가능성, 테스트, 리뷰, 릴리스, 문서화 과정을 함께 훈련하는 것이 목적이다.

사용자는 코드 개발자로 직접 참여한다. 동시에 코드 개발자 2명, 도메인 전문가, 기획자, QA, 릴리스 관리자 역할을 하네스 형태로 구성하여 각 작업을 다각도로 검증한다. 이 과정에서 생성되는 문서, 판단 근거, 리뷰 기록, 테스트 결과, 릴리스 기록이 포트폴리오 증거물이 된다.

## 2. 사용자와 목표

| 사용자 유형 | 목표 | 주요 행동 |
| --- | --- | --- |
| 사용자 개발자 | 문서를 기준으로 구현 방향과 품질을 검증한다 | 스펙 검토, 코드 작성, 리뷰 참여, 최종 판단 |
| 기획자 역할 | 기능 목적과 수용 기준을 명확히 한다 | Issue 작성, 사용자 시나리오 정리, 범위 정의 |
| 도메인 전문가 역할 | 금융 도메인 규칙과 불변식을 검토한다 | 도메인 정책, 예외 케이스, 위험 요소 점검 |
| 코드 개발자 A | 기능을 구현하고 테스트를 작성한다 | 코드 작성, 단위 테스트, 통합 테스트, PR 설명 |
| 코드 개발자 B | 대안과 결함을 검토한다 | 코드 리뷰, 설계 반박, 개선안 제시 |
| QA 역할 | 완료 기준과 회귀 위험을 검증한다 | QA 체크리스트, 경계값, 실패 시나리오 점검 |
| 릴리스 관리자 역할 | 실행 가능한 결과물을 버전으로 묶는다 | 릴리스 노트, 태그, 실행 방법, 알려진 리스크 정리 |

## 3. 사용 시나리오

### 정상 흐름

1. 사용자가 학습하거나 구현할 기능 주제를 선택한다.
2. 기획자 역할이 Issue 수준의 문제 정의와 수용 기준을 만든다.
3. 도메인 전문가 역할이 금융 규칙, 불변식, 위험 요소를 보강한다.
4. 사용자와 개발자 역할이 기능 브랜치를 기준으로 코드를 작성한다.
5. 코드 개발자 A/B가 서로 다른 관점으로 리뷰한다.
6. QA 역할이 테스트와 체크리스트를 기준으로 완료 여부를 검증한다.
7. 중요한 결정은 ADR로 남기고, 전체 사고 과정은 Wiki에 기록한다.
8. 기능이 실행 가능하면 릴리스 노트와 함께 버전으로 묶는다.

### 예외 상황

| 상황 | 사용자에게 보이는 것 | 시스템 동작 |
| --- | --- | --- |
| 요구사항이 불명확함 | 구현 전에 질문 또는 인터뷰가 진행됨 | 스펙 문서를 먼저 보강하고 코드 작업을 보류 |
| 기술 선택 근거가 부족함 | ADR 작성이 요구됨 | 대안, 트레이드오프, 결정 근거를 문서화 |
| 테스트가 없음 | PR 완료 불가 | 테스트 작성 전까지 완료 상태로 보지 않음 |
| 문서와 코드가 불일치함 | 검증 실패로 표시됨 | README, ADR, Wiki, PR 설명 중 기준 문서를 갱신 |
| 릴리스 실행 방법이 없음 | 릴리스 완료 불가 | 로컬 실행 또는 간단한 배포 검증 절차를 추가 |

## 4. 데이터와 상태

### 주요 데이터

| 항목 | 설명 | 필수 여부 | 제약 |
| --- | --- | --- | --- |
| README | 저장소 목적, 배경, 운영 원칙, 진입점 | 필수 | 상세 결정 대신 링크와 요약 중심 |
| ADR | 중요한 기술, 운영, 도메인 결정의 기준 문서 | 필수 | 결정 배경, 대안, 트레이드오프, 결과 포함 |
| Wiki | 역할별 사고 과정, 도메인 설명, 긴 논의 로그 | 필수 | 전체 사고 과정을 남기되 PR에는 요약 링크 |
| Issue | 작업의 입력 단위 | 필수 | 문제, 범위, 수용 기준, 역할 체크 포함 |
| PR | 변경 검증 단위 | 필수 | 코드, 테스트, 리뷰 요약, QA 결과 포함 |
| Release | 완료된 결과물 묶음 | 필수 | 테스트 결과, 실행 방법, 알려진 리스크 포함 |
| Test Result | 기능 완료의 근거 | 필수 | 단위/통합/회귀 테스트 결과를 추적 |
| Role Review | 하네스 역할별 판단 기록 | 필수 | 초기에는 상세 기록, 안정 후 체크리스트화 |

## 6. 권한

| 역할 | 할 수 있는 것 | 할 수 없는 것 |
| --- | --- | --- |
| 사용자 개발자 | 최종 결정, 코드 작성, 리뷰 승인, 방향 변경 | 테스트와 문서 없는 완료 승인 |
| 기획자 역할 | 요구사항, 수용 기준, 범위 제안 | 기술 결정 단독 확정 |
| 도메인 전문가 역할 | 금융 규칙과 위험 요소 지적 | 구현 편의만으로 도메인 규칙 완화 |
| 코드 개발자 A | 구현과 테스트 작성 | 리뷰 없이 병합 |
| 코드 개발자 B | 코드 리뷰와 대안 제시 | 근거 없는 스타일 변경 강제 |
| QA 역할 | 테스트 시나리오와 실패 조건 제시 | 도메인 정책 자체 결정 |
| 릴리스 관리자 역할 | 릴리스 노트와 실행 검증 정리 | 실패한 테스트를 무시하고 릴리스 완료 처리 |

## 7. 범위와 성공 기준

### 이번에 하는 것

- 운영 하네스와 포트폴리오형 레포의 기준을 문서로 고정한다.
- README, ADR, Wiki, Issue/PR 템플릿의 역할 경계를 명확히 한다.
- 하네스 역할별 산출물과 검증 기준을 정의한다.
- 첫 기능 흐름을 `잔액/거래내역 → 회원/지갑 계정 → 충전/송금` 순서로 둔다.
- 문서, 코드, 테스트, 릴리스가 모두 완료 기준에 포함되도록 한다.

### 이번에 안 하는 것

- 실제 금융 거래망, 실명 인증, 외부 결제망 연동은 포함하지 않는다.
- 실제 고객 개인정보 처리는 포함하지 않는다.
- 모든 보안/인증 정책을 첫 단계에서 완성하지 않는다.
- Fly.io 등 외부 배포는 즉시 필수로 두지 않고 릴리스 검증 후보로 둔다.

### 성공 기준

| 기준 | 완료 판단 |
| --- | --- |
| 문서 품질 | 사용자가 README, ADR, Wiki, 템플릿만 보고 작업 흐름을 검증할 수 있음 |
| 결정 근거 | 중요한 결정이 ADR에 대안과 트레이드오프로 기록됨 |
| 사고 과정 | 역할별 판단과 논의 과정이 Wiki 중심으로 추적 가능함 |
| 기능 산출물 | 기능 작업마다 코드와 테스트 코드가 함께 있음 |
| 리뷰 산출물 | 개발자 A/B, 도메인, QA 관점의 검토 결과가 남음 |
| 릴리스 산출물 | 릴리스 노트, 테스트 결과, 실행 방법, 알려진 리스크가 있음 |
| 실행 검증 | 최소 로컬 환경에서 실행 가능하고, 가능하면 간단 배포 환경까지 검토됨 |

## 8. 개발 정책

| 결정 항목 | 정책 | 근거 |
| --- | --- | --- |
| 문서 권위 | 중요한 결정의 source of truth는 ADR로 둔다 | 결정 근거와 트레이드오프를 가장 안정적으로 추적하기 위함 |
| 사고 과정 기록 | 전체 사고 과정은 Wiki 중심으로 남긴다 | 포트폴리오 설명력과 학습 회고 가치를 높이기 위함 |
| README 역할 | README는 목적, 배경, 운영 원칙, 진입점 중심으로 유지한다 | README가 과도하게 길어지는 것을 막고 상세 문서는 Wiki/ADR로 분리 |
| Issue 역할 | 작업 전 문제 정의, 범위, 수용 기준, 역할 체크를 담는다 | 작업 착수 전 모호성을 줄이기 위함 |
| PR 역할 | 변경 요약, 테스트 결과, 리뷰 요약, Wiki/ADR 링크를 담는다 | 코드 변경과 검증 근거를 함께 추적하기 위함 |
| 테스트 정책 | 기능 코드는 테스트 코드 없이 완료될 수 없다 | 금융 도메인은 정합성과 회귀 방지가 핵심이기 때문 |
| 릴리스 정책 | 릴리스는 실행 방법과 알려진 리스크를 포함해야 한다 | 단순 태그가 아니라 검증 가능한 결과물이어야 하기 때문 |
| 하네스 강도 | 초기에는 모든 역할을 촘촘히 적용하고, 반복 패턴이 안정되면 체크리스트 중심으로 경량화한다 | 학습 초기에는 사고 과정을 드러내고, 이후 운영 비용을 줄이기 위함 |
| 첫 기능 순서 | `잔액/거래내역 → 회원/지갑 계정 → 충전/송금` 순서로 진행한다 | 보여지는 금융 기능에서 시작해 기반 모델과 돈 이동 난도로 확장하기 위함 |
| 미결 결정 | 기술 버전, MCP 연결, 외부 배포, 실제 보안 범위는 DECIDE_LATER로 둔다 | 지금 확정하면 근거보다 추정이 커질 수 있으므로 ADR에서 별도 결정 |

## 미결 사항

- Spring Boot 4.x와 3.5.x 중 최종 선택.
- Gradle 9.x와 8.14+ 중 최종 선택.
- GitHub MCP 실제 연결 시점과 권한 범위.
- PostgreSQL MCP 도입 시점.
- Fly.io 같은 외부 배포를 필수 릴리스 조건으로 올릴 시점.
- 실제 금융권 수준 보안, 인증, 개인정보 보호를 다루기 시작할 릴리스.
