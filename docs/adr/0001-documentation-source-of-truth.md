# ADR-0001: Documentation Source of Truth

## 상태

Accepted

## 맥락

이 저장소는 Java 25와 Spring Boot 기반 핀테크 학습 레포이면서, 운영 하네스와 포트폴리오형 결과물을 함께 목표로 한다.

문서의 종류가 많다.

- `README.md`: 저장소 진입점
- `docs/adr`: 중요한 결정 기록
- GitHub Wiki: 긴 도메인 설명과 사고 과정
- GitHub Issue: 작업 입력
- GitHub Pull Request: 변경 검증
- GitHub Release: 버전별 결과물

문서의 책임이 겹치면 어떤 문서가 기준인지 불명확해진다. 특히 금융 도메인 학습에서는 기술 선택, 테스트 전략, 도메인 정책, 릴리스 기준의 근거가 중요하므로 결정의 기준 문서를 명확히 해야 한다.

## 선택지

### 선택지 A: README 중심

README에 모든 핵심 내용을 모으고 다른 문서는 부가 자료로 둔다.

장점:

- 진입점이 단순하다.
- 새 참여자가 한 파일만 읽어도 전체 방향을 파악하기 쉽다.

단점:

- README가 빠르게 비대해진다.
- 상세한 트레이드오프와 사고 과정을 담기 어렵다.
- 결정 변경 이력을 추적하기 어렵다.

### 선택지 B: Wiki 중심

Wiki를 모든 상세 문서와 결정의 기준으로 둔다.

장점:

- 긴 설명, 역할별 사고 과정, 도메인 학습 자료를 담기 좋다.
- 포트폴리오 관점에서 읽기 좋은 문서를 구성하기 쉽다.

단점:

- 코드 변경과 함께 리뷰하기 어렵다.
- GitHub Wiki 변경은 PR 검증 흐름과 분리될 수 있다.
- 결정 자체와 논의 로그가 섞일 위험이 있다.

### 선택지 C: ADR 중심

중요한 결정의 source of truth를 ADR로 두고, README와 Wiki는 ADR을 요약하거나 연결한다.

장점:

- 결정 배경, 선택지, 트레이드오프, 결과를 일관된 형식으로 남길 수 있다.
- 코드와 함께 PR에서 리뷰할 수 있다.
- 결정 변경 시 새 ADR로 대체 관계를 추적할 수 있다.
- README와 Wiki의 책임을 분리하기 쉽다.

단점:

- ADR 작성 비용이 추가된다.
- 모든 문서 작성자가 ADR과 Wiki의 경계를 이해해야 한다.

## 결정

중요한 결정의 source of truth는 `docs/adr`에 둔다.

문서별 책임은 다음과 같이 나눈다.

| 문서 | 책임 | 기준 여부 |
| --- | --- | --- |
| `README.md` | 목적, 배경, 운영 원칙, 빠른 진입점 | 요약 |
| `docs/adr` | 기술, 운영, 도메인 정책의 중요 결정 | 기준 |
| GitHub Wiki | 긴 설명, 역할별 사고 과정, 도메인 학습 로그 | 상세 기록 |
| GitHub Issue | 작업 전 문제 정의, 범위, 수용 기준 | 작업 입력 |
| GitHub PR | 변경 요약, 테스트 결과, 리뷰 요약, ADR/Wiki 링크 | 변경 검증 |
| GitHub Release | 완료 기능, 실행 방법, 테스트 결과, 알려진 리스크 | 버전 결과 |

Wiki는 전체 사고 과정을 남기는 공간으로 사용한다. 단, 결정 그 자체는 ADR에 고정한다. PR은 Wiki와 ADR의 링크를 포함하고, 변경 검증 결과를 요약한다.

## 결과

장점:

- 중요한 판단의 근거가 코드 저장소 안에 남는다.
- README는 진입점 역할에 집중할 수 있다.
- Wiki는 포트폴리오형 설명과 사고 과정 기록에 집중할 수 있다.
- PR 리뷰에서 결정 근거까지 함께 검증할 수 있다.

비용:

- 기능 작업마다 ADR 필요 여부를 판단해야 한다.
- Wiki와 ADR의 중복을 지속적으로 관리해야 한다.

후속 작업:

- Issue 템플릿에 ADR 필요 여부 항목을 추가한다.
- PR 템플릿에 관련 ADR/Wiki 링크 항목을 추가한다.
- Wiki 초안에는 ADR 링크를 포함한다.
