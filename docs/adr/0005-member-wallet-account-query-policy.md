# ADR-0005: Member Wallet Account Query Policy

## 상태

Accepted

## 맥락

Issue #1의 잔액/거래내역 조회 API는 `wallet-001` 샘플 데이터를 기준으로 빠르게 기능 흐름을 검증했다. Issue #3에서는 조회 기능을 회원/지갑 계정 모델 위로 올려야 한다.

초기 범위에서 해결해야 하는 질문은 다음과 같다.

- 지갑이 어떤 회원에게 속하는지 코드와 문서에서 표현할 수 있는가?
- 회원 또는 지갑이 비활성 상태일 때 잔액/거래내역 조회를 허용할 것인가?
- 복수 지갑 가능성을 열어두면서도 첫 기능 구현을 과도하게 복잡하게 만들지 않을 수 있는가?

핀테크 서비스에서 계정 생명주기는 돈 이동, 고객 보호, 감사 추적과 직접 연결된다. 따라서 조회 기능이라도 소유 관계와 상태 정책을 생략하면 이후 충전/송금 기능의 검증 기준이 약해진다.

## 선택지

### 선택지 A: 지갑 식별자만 유지

기존처럼 지갑 식별자만으로 잔액과 거래내역을 조회한다.

장점:

- 구현이 가장 단순하다.
- 첫 조회 API의 코드 변경이 적다.
- 인증/인가 도입 전까지 빠르게 진행할 수 있다.

단점:

- 지갑 소유 관계가 코드에 드러나지 않는다.
- 회원 정지, 탈퇴, 지갑 해지 같은 상태 정책을 표현하기 어렵다.
- 충전/송금 단계에서 권한과 계정 상태 검증을 뒤늦게 끼워 넣어야 한다.

### 선택지 B: 회원당 기본 지갑 1개로 고정

회원과 지갑을 1:1 관계로 모델링하고, 회원 식별자를 중심으로 조회한다.

장점:

- 사용자 관점의 기본 계좌/기본 지갑 흐름을 단순하게 표현할 수 있다.
- 조회 API의 소유자 개념이 명확해진다.
- 초기 학습 범위가 작다.

단점:

- 복수 지갑, 목적별 지갑, 통화별 지갑 확장 시 모델 변경 비용이 생긴다.
- 지갑 식별자 기반 거래내역/원장 연결이 약해질 수 있다.
- 실제 핀테크 서비스의 계좌/지갑 다양성을 과도하게 단순화한다.

### 선택지 C: 회원은 복수 지갑 가능, 초기 API는 기본 지갑 흐름

도메인 모델은 회원이 0개 이상의 지갑을 가질 수 있게 열어두고, 초기 샘플 데이터와 API는 단일 기본 지갑 흐름으로 제한한다. 조회 가능 조건은 `회원 ACTIVE`와 `지갑 ACTIVE`를 모두 만족해야 한다.

장점:

- 이후 복수 지갑과 충전/송금 확장 여지를 보존한다.
- 초기 구현은 단일 샘플 흐름으로 유지해 복잡도를 낮춘다.
- 계정 상태 정책을 조회 단계부터 테스트할 수 있다.
- 권한/인증 도입 전에도 소유 관계와 생명주기 검증 기준을 남길 수 있다.

단점:

- 지갑 조회 시 회원 저장소 조회가 추가된다.
- 실제 인증 주체와 회원 식별자 연결은 아직 별도 결정이 필요하다.
- 비활성 상태별 세부 정책은 추가 ADR 또는 Wiki 결정이 필요하다.

## 결정

`회원은 복수 지갑 가능, 초기 API는 기본 지갑 흐름`을 선택한다.

초기 구현 기준은 다음과 같다.

| 항목 | 결정 |
| --- | --- |
| 회원 모델 | `Member`로 표현하고 `memberId`, `status`, `createdAt`을 가진다 |
| 지갑 모델 | `WalletAccount`로 표현하고 `walletId`, `memberId`, `status`, `createdAt`을 가진다 |
| 회원 상태 | `ACTIVE`, `SUSPENDED`, `CLOSED` 후보를 둔다 |
| 지갑 상태 | `ACTIVE`, `SUSPENDED`, `CLOSED` 후보를 둔다 |
| 조회 가능 조건 | 회원과 지갑이 모두 `ACTIVE`일 때만 잔액/거래내역 조회 가능 |
| 초기 샘플 | `member-001`이 `wallet-001`을 소유하는 단일 기본 지갑으로 시작 |
| 비활성 응답 | 초기 API에서는 `409 CONFLICT`와 `WALLET_NOT_QUERYABLE`로 표현 |

## 결과

장점:

- 잔액/거래내역 조회가 회원/지갑 계정 모델 위에서 동작한다.
- `SUSPENDED`, `CLOSED` 상태가 테스트 가능한 도메인 규칙으로 들어온다.
- 충전/송금에서 사용할 소유 관계와 계정 상태 검증의 출발점이 생긴다.

비용:

- 조회 서비스가 지갑 저장소뿐 아니라 회원 저장소 계약에도 의존한다.
- 인증/인가가 없으므로 아직 “요청자와 회원이 같은가”는 검증하지 않는다.
- 비활성 회원/지갑의 거래내역 열람, 탈퇴 보존 정책은 후속 결정으로 남는다.

후속 작업:

- 충전/송금 기능에서 동일한 계정 상태 검증 기준을 재사용한다.
- 회원 탈퇴 시 지갑과 거래내역 보존 정책을 별도 ADR로 결정한다.
- 인증/인가 도입 시 요청 주체와 `memberId` 매핑 정책을 결정한다.
