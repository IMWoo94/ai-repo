# ADR-0024: React User Frontend MVP

## 상태

Accepted

## 맥락

현재 백엔드는 잔액/거래내역, 충전/송금, 원장, 감사 로그, step log, outbox 조회 API를 제공한다. 다음 단계에서는 사용자가 API 호출 결과를 직접 확인할 수 있는 화면이 필요하다.

요구사항은 다음과 같다.

- React 기반으로 사용자 화면을 만든다.
- UI 생성 기준은 `DESIGN-apple.md`를 따른다.
- 현재는 모놀리식 백엔드지만, 향후 MSA 전환을 고려해 프론트와 백엔드 실행 경계를 분리한다.
- 코드뿐 아니라 실행/검증 문서와 CI 게이트를 함께 남긴다.

## 선택지

### 선택지 A: Spring Boot 정적 HTML로 시작한다

장점:

- 빌드 구성이 가장 단순하다.
- 백엔드 서버 하나만 실행하면 된다.

단점:

- React 학습과 컴포넌트 기반 UI 확장 목적에 맞지 않는다.
- API 서버와 사용자 화면의 변경 주기가 강하게 묶인다.

### 선택지 B: React와 Vite를 별도 프론트 앱으로 둔다

장점:

- React 컴포넌트, 상태 관리, API 호출 흐름을 명확히 학습할 수 있다.
- Vite 개발 서버 proxy로 로컬 백엔드와 빠르게 연결할 수 있다.
- 향후 MSA/API Gateway 도입 시 프론트 배포 단위를 분리하기 쉽다.

단점:

- Node/npm 의존성과 별도 빌드 게이트가 추가된다.
- 백엔드와 프론트 실행 명령이 분리된다.

### 선택지 C: Next.js로 시작한다

장점:

- 라우팅, 서버 컴포넌트, 배포 생태계가 풍부하다.
- 포트폴리오형 화면 확장에 유리하다.

단점:

- 현재 범위에는 SSR, 파일 기반 라우팅, 서버 액션이 필요하지 않다.
- 금융 도메인 백엔드 학습보다 프론트 프레임워크 복잡도가 앞설 수 있다.

## 결정

React와 Vite를 별도 `frontend` 앱으로 둔다.

구현 기준은 다음과 같다.

| 항목 | 결정 |
| --- | --- |
| 프레임워크 | React |
| 번들러/개발 서버 | Vite |
| 언어 | TypeScript |
| 로컬 연결 | Vite proxy로 `/api`를 `localhost:8080`에 전달 |
| 디자인 기준 | `DESIGN-apple.md` |
| CI 게이트 | `npm ci` 후 `npm run build` |

## 결과

장점:

- 사용자 화면에서 지갑 조회, 충전, 송금, 원장/감사/운영 증거 확인 흐름을 한 번에 검증할 수 있다.
- UI 규칙이 `AGENTS.md`와 문서에 명시되어 이후 화면 추가 시 기준이 유지된다.
- 백엔드 Gradle 게이트와 프론트 npm build 게이트를 분리해 실패 원인을 명확히 볼 수 있다.

비용:

- Node 런타임과 npm lockfile 관리가 추가된다.
- 프론트 테스트는 아직 build smoke 수준이다.

후속 작업:

- 프론트 단위/컴포넌트 테스트 도입 여부를 별도 ADR로 결정한다.
- 운영자용 outbox manual review 화면은 다음 UI 단계에서 분리한다.
- 실제 배포는 정적 호스팅, Spring resource packaging, 별도 컨테이너 중 하나로 후속 결정한다.
