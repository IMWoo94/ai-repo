# ADR-0007: Ledger Audit Log Boundary

## 상태

Accepted

## 맥락

충전/송금 1차 구현은 잔액 변경과 사용자 거래내역을 남긴다. 하지만 거래내역은 사용자가 이해할 수 있는 표시용 기록이고, 원장은 잔액 정합성을 검증하기 위한 시스템 기록이다. 감사 로그는 원장과도 다르게 “어떤 명령이 처리되었는가”를 추적한다.

Issue #7의 목표는 PostgreSQL 도입 전, 이 세 책임의 경계를 코드와 문서로 먼저 고정하는 것이다.

## 선택지

### 선택지 A: 거래내역을 원장으로 계속 사용

거래내역 모델에 잔액 변경 검증과 사용자 표시 책임을 모두 둔다.

장점:

- 구현 변경이 작다.
- 조회 API와 저장소 모델이 단순하다.
- 초기 기능 속도가 빠르다.

단점:

- 사용자 표시 요구가 바뀌면 정합성 검증 기록도 흔들릴 수 있다.
- 원장 불변성, 보정, 재처리 정책을 분리하기 어렵다.
- 감사 추적과 잔액 검증의 목적이 섞인다.

### 선택지 B: 원장과 감사 로그를 별도 모델로 분리

거래내역은 사용자 조회용, 원장은 잔액 변경 검증용, 감사 로그는 명령 처리 추적용으로 분리한다.

장점:

- 각 기록의 목적과 검증 기준이 명확해진다.
- PostgreSQL 테이블과 제약 설계의 기준점이 생긴다.
- 멱등 재시도 시 중복 원장 생성 방지를 테스트할 수 있다.

단점:

- 인메모리 저장소가 기록을 더 많이 관리한다.
- 아직 DB 수준 append-only 제약은 없다.
- API 접근 제어는 후속 인증/인가 결정이 필요하다.

## 결정

`원장과 감사 로그를 별도 모델로 분리`한다.

초기 구현 기준은 다음과 같다.

| 기록 | 목적 | 초기 API |
| --- | --- | --- |
| 거래내역 | 사용자에게 보여줄 금융 활동 기록 | `GET /api/v1/wallets/{walletId}/transactions` |
| 원장 엔트리 | 잔액 변경과 변경 후 잔액 검증 기록 | `GET /api/v1/wallets/{walletId}/ledger-entries` |
| 감사 이벤트 | 명령 처리 결과 추적 기록 | `GET /api/v1/audit-events` |

성공한 충전/송금은 거래내역, 원장 엔트리, 감사 이벤트를 함께 남긴다. 같은 멱등키와 같은 요청의 재시도는 기존 결과를 반환하고 원장/감사 기록을 중복 생성하지 않는다.

## 결과

장점:

- 원장 모델이 PostgreSQL 영속화 전 단계에서 먼저 검증된다.
- 잔액 변경 후 잔액이 원장에 남아 정합성 검증 근거가 된다.
- 감사 이벤트가 명령 처리 추적의 독립 증거가 된다.

비용:

- 인메모리 저장소의 책임이 늘어난다.
- 원장/감사 조회 API는 아직 인증 없이 노출된다.
- 실패, 취소, 보상, 재처리 원장 정책은 후속 결정이 필요하다.

후속 작업:

- PostgreSQL 도입 시 원장 append-only 제약과 인덱스를 설계한다.
- 감사 로그 접근 권한과 마스킹 정책을 인증/인가 ADR에서 결정한다.
- 실패/취소/보상 거래의 원장 표현 방식을 별도 ADR로 결정한다.
