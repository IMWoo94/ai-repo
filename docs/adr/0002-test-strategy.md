# ADR-0002: Test Strategy

## 상태

Accepted

## 맥락

이 저장소의 목표는 핀테크 서비스를 학습하는 것이다. 핀테크 도메인은 기능이 단순해 보여도 데이터 정합성, 동시성, 감사 가능성, 장애 복구, 중복 요청 방지가 핵심이다.

사용자는 문서를 기준으로 구현을 검증하고, 기능 작업은 코드와 테스트 코드를 필수로 포함하기로 합의했다. 따라서 테스트 전략은 단순한 품질 보조 수단이 아니라 작업 완료 조건이다.

초기 기능 흐름은 다음 순서로 진행한다.

1. 잔액과 거래내역
2. 회원과 지갑 계정
3. 충전과 송금

이 순서는 조회 중심 기능에서 시작해 계정 기반 모델을 정리하고, 이후 돈 이동과 정합성 문제로 확장하기 위한 것이다.

## 선택지

### 선택지 A: 단위 테스트 중심

도메인 객체와 서비스 로직을 빠른 단위 테스트 위주로 검증한다.

장점:

- 빠르게 작성하고 실행할 수 있다.
- 도메인 규칙과 상태 전이를 좁은 범위에서 검증하기 좋다.
- 초기 개발 속도가 빠르다.

단점:

- DB 트랜잭션, 제약 조건, 동시성 문제를 충분히 검증하기 어렵다.
- API 경계와 실제 실행 환경의 오류를 놓칠 수 있다.

### 선택지 B: 통합 테스트 중심

Spring Boot와 DB를 포함한 통합 테스트를 중심으로 검증한다.

장점:

- 실제 애플리케이션 경계와 데이터 저장 동작을 검증할 수 있다.
- Repository, 트랜잭션, API 요청/응답 오류를 발견하기 좋다.

단점:

- 실행 시간이 길어진다.
- 실패 원인 분석 비용이 증가한다.
- 도메인 규칙의 세밀한 실패를 표현하기 어렵다.

### 선택지 C: 테스트 피라미드와 위험 기반 테스트 결합

단위 테스트를 기본으로 두고, 통합 테스트와 동시성/회귀 테스트를 위험도에 따라 추가한다.

장점:

- 빠른 피드백과 실제 경계 검증을 함께 확보할 수 있다.
- 금융 도메인의 고위험 기능에 테스트 비용을 집중할 수 있다.
- 기능별 완료 기준을 명확히 만들 수 있다.

단점:

- 테스트 유형별 책임을 문서화해야 한다.
- 초기 설정 비용이 단위 테스트 중심보다 높다.

## 결정

테스트 전략은 `테스트 피라미드와 위험 기반 테스트 결합`으로 한다.

기능 코드는 테스트 코드 없이 완료될 수 없다. 기능의 위험도에 따라 필요한 테스트 수준을 다르게 적용한다.

| 테스트 유형 | 목적 | 적용 기준 |
| --- | --- | --- |
| 단위 테스트 | 도메인 규칙, 계산, 상태 전이 검증 | 모든 기능 |
| 통합 테스트 | API, Repository, 트랜잭션, DB 제약 검증 | 저장소 또는 API 경계가 있는 기능 |
| 동시성 테스트 | 잔액 변경, 중복 요청, 락 전략 검증 | 충전, 송금, 원장 등 돈 이동 기능 |
| 회귀 테스트 | 재발 방지 | 버그 수정 시 실패 재현 먼저 작성 |
| 실행 검증 | 릴리스 후보가 실제로 실행 가능한지 확인 | 릴리스 대상 변경 |

초기 품질 게이트는 다음 명령을 기준으로 한다.

```bash
./gradlew test
./gradlew check
```

프로젝트 스캐폴딩 전에는 위 명령이 준비 예정 상태다. 스캐폴딩이 추가되면 CI와 로컬 검증 명령으로 승격한다.

## 기능별 최소 테스트 기준

| 기능 범위 | 최소 테스트 | 추가 테스트 |
| --- | --- | --- |
| 잔액 조회 | 단위 테스트, API 통합 테스트 | 잘못된 사용자/지갑 조회 |
| 거래내역 조회 | 단위 테스트, API 통합 테스트 | 페이징, 정렬, 빈 결과 |
| 회원/지갑 계정 | 단위 테스트, Repository 통합 테스트 | 중복 생성, 상태 전이 |
| 충전 | 단위 테스트, 통합 테스트 | 멱등 키, 중복 요청, 한도 |
| 송금 | 단위 테스트, 통합 테스트 | 동시성, 잔액 부족, 보상 처리 후보 |
| 원장/감사 로그 | 단위 테스트, 통합 테스트 | 불변성, 수정 불가 정책 |

## 완료 기준

PR은 다음을 만족해야 한다.

- 변경된 도메인 규칙에 대응하는 단위 테스트가 있다.
- 저장소나 API 경계가 있으면 통합 테스트가 있다.
- 돈 이동, 잔액 변경, 중복 요청 관련 기능이면 동시성 또는 멱등성 테스트 계획이 있다.
- 버그 수정이면 실패를 재현하는 회귀 테스트가 먼저 있다.
- 테스트 실행 결과를 PR에 기록한다.
- 릴리스 대상이면 로컬 실행 검증 결과를 릴리스 노트에 기록한다.

## 결과

장점:

- 테스트가 기능 완료 조건으로 명확해진다.
- 빠른 단위 테스트와 실제 경계 검증을 균형 있게 가져갈 수 있다.
- 핀테크 핵심 위험인 정합성, 중복 요청, 동시성을 별도 검증 대상으로 둘 수 있다.

비용:

- 초기 프로젝트 설정과 테스트 인프라 구성이 필요하다.
- 통합 테스트와 동시성 테스트는 실행 시간이 늘어날 수 있다.
- 테스트 작성 기준을 PR 리뷰에서 지속적으로 확인해야 한다.

후속 작업:

- PR 템플릿에 테스트 유형별 체크박스를 추가한다.
- Issue 템플릿에 예상 테스트 범위를 추가한다.
- 첫 Spring Boot 스캐폴딩에서 JUnit 5, Spring Boot Test, Testcontainers 도입 여부를 별도 ADR로 결정한다.
