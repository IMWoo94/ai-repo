# ADR-0011: PostgreSQL Balance Row Locking

## 상태

Accepted

## 맥락

충전과 송금은 잔액, 거래내역, 원장, 감사 로그, 멱등 기록을 하나의 정합성 단위로 다룬다. 인메모리 구현은 서비스 메서드의 `synchronized`로 단일 JVM 내부 동시 실행을 막지만, PostgreSQL 저장소는 여러 애플리케이션 인스턴스 또는 여러 서비스 객체가 같은 DB를 동시에 갱신할 수 있다.

기존 PostgreSQL 저장소는 서비스 계층에서 잔액 부족을 확인한 뒤 저장소 트랜잭션에서 잔액을 갱신했다. 이 방식은 두 송금 요청이 같은 출금 지갑 잔액을 동시에 읽으면 잔액 음수 방지와 원장 정합성을 DB 트랜잭션 안에서 보장하지 못한다.

## 선택지

### 선택지 A: 서비스 `synchronized` 유지

장점:

- 코드가 단순하다.
- 단일 프로세스 학습용 실행에서는 효과가 있다.

단점:

- 여러 JVM 또는 여러 서비스 인스턴스가 같은 DB를 갱신하는 상황을 보호하지 못한다.
- 금융 잔액 불변식이 DB 트랜잭션이 아니라 애플리케이션 객체 락에 의존한다.

### 선택지 B: PostgreSQL row-level lock

장점:

- 잔액 불변식을 DB 트랜잭션 내부에서 보호한다.
- 갱신 대상 지갑 row만 잠가 lock 범위를 작게 유지한다.
- 기존 스키마 변경 없이 적용할 수 있다.

단점:

- 같은 지갑에 송금이 몰리면 lock 대기가 생긴다.
- 두 지갑 송금에서는 lock 획득 순서를 고정하지 않으면 deadlock 위험이 있다.
- 장기 트랜잭션이 생기면 다른 잔액 변경 요청을 지연시킬 수 있다.

### 선택지 C: 낙관적 락 version column

장점:

- 충돌이 드문 환경에서는 대기보다 재시도가 효율적일 수 있다.
- 충돌 감지와 재시도 정책을 명시적으로 만들 수 있다.

단점:

- 스키마 변경이 필요하다.
- 충돌 재시도 정책, 응답 코드, 사용자 경험을 함께 설계해야 한다.
- 현재 단계에서는 row-level lock보다 구현 범위가 크다.

### 선택지 D: 전역 `SERIALIZABLE` 격리 수준

장점:

- 넓은 범위의 동시성 이상 현상을 DB가 감지한다.
- 코드에서 개별 lock을 덜 명시할 수 있다.

단점:

- 전체 트랜잭션 비용과 재시도 가능성이 커진다.
- 현재 잔액 row 단위 문제에는 과한 선택이다.

## 결정

`PostgreSQL row-level lock`을 선택한다.

초기 구현 기준은 다음과 같다.

| 항목 | 결정 |
| --- | --- |
| 충전 | 대상 지갑의 `wallet_balances` row를 `SELECT ... FOR UPDATE`로 잠근 뒤 갱신 |
| 송금 | 출금/입금 지갑의 `wallet_balances` row를 결정적 순서로 잠근 뒤 갱신 |
| 잔액 부족 검증 | 서비스 사전 검증에 더해 저장소 트랜잭션 내부 locked balance 기준으로 재검증 |
| 실패한 송금 | 거래내역, 원장, 감사 로그, 멱등 기록을 남기지 않음 |
| 테스트 | Testcontainers PostgreSQL 동시 송금 테스트로 최종 잔액과 기록 수 검증 |

## 결과

장점:

- 잔액 음수 방지 불변식이 DB 트랜잭션 내부로 이동한다.
- 동시 송금에서 성공한 요청만 원장/감사 로그를 남긴다.
- 스키마 변경 없이 운영 유사 동시성 검증을 추가한다.

비용:

- 같은 지갑에 대한 충전/송금 요청은 직렬화된다.
- lock 대기와 timeout 정책은 아직 별도 운영 정책으로 남아 있다.
- Docker가 없는 로컬 환경에서는 Testcontainers 동시성 검증이 스킵될 수 있다.

후속 작업:

- lock wait timeout과 사용자 응답 정책을 정의한다.
- 낙관적 락 version column 도입 여부를 재검토한다.
- 송금 실패 이력을 사용자 거래내역에 남길지 결정한다.
