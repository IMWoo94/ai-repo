# ADR-0006: Charge Transfer Idempotency Policy

## 상태

Accepted

## 맥락

충전과 송금은 이 저장소의 첫 잔액 변경 기능이다. 조회 기능과 달리 같은 요청이 두 번 처리되면 잔액이 잘못 증가하거나 감소한다. 특히 모바일 네트워크 재시도, 사용자의 중복 클릭, 서버 타임아웃 후 재요청은 핀테크 서비스에서 일반적인 실패 모드다.

Issue #5의 1차 목표는 실제 외부 연동이나 DB 트랜잭션까지 구현하는 것이 아니라, 잔액 변경 기능의 최소 안전장치를 코드와 테스트로 고정하는 것이다.

## 선택지

### 선택지 A: 멱등성 없이 단순 처리

충전과 송금 요청이 들어올 때마다 잔액을 변경한다.

장점:

- 구현이 가장 단순하다.
- API와 저장소 코드가 작다.
- 초기 데모를 빠르게 만들 수 있다.

단점:

- 동일 요청 재시도 시 잔액이 중복 반영된다.
- 장애 상황을 학습하기 어렵다.
- 이후 멱등키를 추가할 때 API 계약과 저장소 책임을 다시 바꿔야 한다.

### 선택지 B: 요청 멱등키를 1차부터 필수로 둔다

충전/송금 요청에 `idempotencyKey`를 필수로 받고, 같은 키와 같은 요청은 기존 결과를 반환한다. 같은 키로 다른 요청이 오면 충돌로 처리한다.

장점:

- 중복 반영 위험을 초기부터 테스트할 수 있다.
- API 사용자가 재시도 가능한 계약을 갖는다.
- 이후 DB 저장소로 이전할 때 유니크 키와 트랜잭션 경계 설계가 명확해진다.

단점:

- 요청 fingerprint 계산과 저장소 기록이 추가된다.
- 멱등키 보관 기간, 충돌 정책 같은 후속 결정이 필요하다.
- 인메모리 저장소에서는 프로세스 재시작 후 멱등성 기록이 사라진다.

### 선택지 C: 거래 원장 도입 후 멱등성을 구현한다

이번 단계에서는 충전/송금만 구현하고, 원장 모델을 만들 때 멱등성을 함께 처리한다.

장점:

- 멱등성 기록과 원장 기록을 한 번에 설계할 수 있다.
- 저장소 영속화 이후 더 현실적인 제약으로 구현할 수 있다.

단점:

- 그 전까지 중복 요청 위험이 코드에 남는다.
- 충전/송금 테스트가 실제 핀테크 위험을 충분히 다루지 못한다.
- 원장 단계에서 변경 범위가 커진다.

## 결정

`요청 멱등키를 1차부터 필수로 둔다`를 선택한다.

초기 구현 기준은 다음과 같다.

| 항목 | 결정 |
| --- | --- |
| 충전 API | `POST /api/v1/wallets/{walletId}/charges` |
| 송금 API | `POST /api/v1/wallets/{walletId}/transfers` |
| 지원 통화 | `KRW` 단일 |
| 처리 방식 | 인메모리 저장소의 동기 처리 |
| 멱등성 | `idempotencyKey` 필수 |
| 같은 키와 같은 요청 | 기존 결과 반환, HTTP 200 |
| 같은 키와 다른 요청 | `409 CONFLICT`, `IDEMPOTENCY_KEY_CONFLICT` |
| 신규 요청 성공 | HTTP 201 |
| 잔액 부족 | `409 CONFLICT`, `INSUFFICIENT_BALANCE` |
| 계정 상태 검증 | 회원과 지갑이 모두 `ACTIVE`여야 함 |

## 결과

장점:

- 충전/송금의 핵심 위험인 중복 반영을 1차부터 검증한다.
- 잔액 변경과 거래내역 기록이 같은 명령 흐름에 묶인다.
- 이후 원장/감사 로그 설계의 기준 입력이 생긴다.

비용:

- 인메모리 저장소가 조회와 명령 책임을 함께 가진다.
- 현재 동기화는 단일 프로세스 범위에서만 유효하다.
- 멱등키 보관 기간, DB 유니크 제약, 동시성 락은 후속 결정이 필요하다.

후속 작업:

- PostgreSQL 도입 시 멱등키 유니크 제약과 트랜잭션 격리 수준을 결정한다.
- 원장 모델을 추가해 잔액 변경과 감사 기록을 분리한다.
- 충전/송금 한도, 수수료, 취소/실패 상태를 별도 정책으로 확장한다.
